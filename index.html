Why?

  • Reduces downtime
  • Reduces errors
  • Consistent, repeatable environments

Approaches

  • State-Based
  • Migration-Based

State-Based

  • Single, canonical definition of schema
  • Engine compares definition to target, performs diff
  • Can also contain "system data" => danger zone

Redgate SQL Toolbelt (demo)

Migration-Based

  • Schema and data are built up from diffs (like git)
  • Special table tracks which migrations have been run
  • Can contain "rollback" functionality

Existing (Framework) Solutions

  • .NET Core
    dotnet ef migrations add
    dotnet ef database update
  • Rails
    bin/rails generate migration
    bin/rails db:migrate
  • Laravel
    php artisan make:migration
    php artisan migrate

Benefits

  • Battle-tested
  • Stronger-typed code
  • Low maintenance

Roll Your Own

  1. Create migrations table if it doesn't exist
  2. Get new migrations
    1. Get all migration scripts
    2. Get completed migrations from DB
    3. Get scripts that aren't in the table
  3. For each new migration (in order):
    1. Run the script
    2. Add the script to the migrations table

Best Practices

  • Small, self-contained changes
  • Prepend filename with deterministic ordering
  • Subsequent/accidental runs should be no-ops
  • Automate updates during deployment
